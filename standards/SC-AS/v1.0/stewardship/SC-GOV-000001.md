# Constitutional Release Procedure

A constitutional release is a distribution of the SC-AS canonical set together with its required stewardship artifacts, certificates, and integrity infrastructure. This section defines the bundle composition, assembly sequence, and approval gate.

## Bundle Composition

A constitutional release bundle MUST contain the following artifacts, organized by governance tier.

**Tier 0 — Canonical Set (SC-SCOPE-000001 §6):**

- SC-CORE-000001
- SC-AXIOM-000001
- SC-HDR-000001
- SC-SCOPE-000001

**Tier 2 — Certificates (SC-SCOPE-000001 §7):**

- One current RCC per canonical document (SC-SCOPE-000001 §7.6)

**Tier 3 — Stewardship Artifacts (SC-SCOPE-000001 §8):**

- SC-GOV-000001 (this document)
- SC-LICENSE-000001
- SC-NAME-000001
- SC-TM-000001
- SC-AUTHORSHIP-000001

**License File:**

- A file named `LICENSE` at the root of the export bundle containing the full text of the Creative Commons Attribution-NoDerivatives 4.0 International License, as specified by SC-LICENSE-000001.

**Release Infrastructure:**

- Release manifest (JSON): file list + SHA-256 + sizes for every artifact in the bundle
- ZIP manifest: SHA-256 for the export ZIP + pointer to the release manifest
- PGP detached signature (ASCII-armored) over the release manifest, signed with a key listed in SC-AUTHORSHIP-000001 §5
- Release notes (NON-NORMATIVE)

## Assembly Sequence

1. Collect all artifacts listed in Bundle Composition above.
2. Run the Conformance Validation Procedure (Phases 1-7) on every artifact with Release Status = ISSUED.
3. Run Phase 8 (Bundle Completeness Verification) on the assembled bundle.
4. Generate the release manifest (JSON) with SHA-256 for every artifact.
5. Generate the export ZIP.
6. Generate the ZIP manifest.
7. Run Phase 9 (Cryptographic Signing) to produce the PGP detached signature over the release manifest.
8. Run the Release Approval Checklist.
9. If all checks pass, publish to canonical publication targets.

# Conformance Validation Procedure (Release Gate)

The following procedure defines the **normative validation checklist** that MUST be satisfied before a release is issued.

Phases 1-7 operationalize the per-artifact schema defined in **SC-HDR-000001**. Phases 8-9 operationalize the bundle-level requirements defined in **SC-SCOPE-000001** (§6, §7, §8) and **SC-AUTHORSHIP-000001** (§5).

---

## Phase 1 — Canonicalization Check

Validator SHALL confirm:

- file encoding is UTF-8
- line endings are LF (`\n`)
- file ends with a final LF
- no trailing whitespace
- no tab characters

Failure of any rule results in **release rejection**.

---

## Phase 2 — Header Surface Detection

Validator SHALL:

1. Locate the heading
   `## Canonical Document Header (Informative / Non-Normative Metadata)`

2. Ignore fenced code blocks when scanning.

3. Treat the first occurrence outside code fences as the canonical header surface.

If the heading is absent → artifact is **non-conformant**.

---

## Phase 3 — Base Header Validation

Validator SHALL confirm presence and order of the following fields:

1. Document ID
2. Canonical Name
3. Document Class
4. Version
5. Issuance ID
6. Release Status
7. Effective Date
8. Last Updated
9. Issuing Authority
10. DOI / Persistent ID
11. Supersedes
12. Superseded By
13. Header Schema
14. Integrity

Rules:

- each field MUST appear exactly once
- no additional base header keys permitted
- order MUST match the canonical sequence

---

## Phase 4 — Field Grammar Validation

Validator SHALL confirm:

Release Status ∈ {DRAFT, ISSUED, DEPRECATED, SUPERSEDED, WITHDRAWN}

Issuance ID grammar:

```
v + two or more digits + optional single lowercase letter
```

Example:

```
v01
v07
v11h
```

---

## Phase 5 — Document Class Extension Validation

Validator SHALL normalize dependency references to the canonical form defined in SC-HDR (Reference Syntax).


Validator SHALL confirm:

- extension block exists
- block appears immediately after base header delimiter (`---`)
- extension heading matches:

```
## Document Class Extension: <CLASS>
```

Extension keys MUST follow:

Validator MUST also recognize Document Class `POLICY` with token `POL` as defined in SC-HDR.

```
<TOKEN>::<Key>: <Value>
```

Where TOKEN matches the document class token defined in SC-HDR.

---

## Phase 6 — Duplicate Metadata Detection

Validator SHALL reject artifacts that contain:

- additional header blocks
- YAML front-matter metadata
- conflicting identity metadata

Filename metadata SHALL NOT be treated as authoritative.

---

## Phase 7 — Integrity Verification

If `Release Status = ISSUED`:

1. Compute SHA-256 over the distributed artifact.
2. Treat the `SHA-256:` value as blank during computation.
3. Compare computed digest with declared digest.

Mismatch → release rejection.

---

## Phase 8 — Bundle Completeness Verification

Validator SHALL confirm that the assembled bundle contains every artifact listed in the Constitutional Release Procedure (Bundle Composition), specifically:

1. All four Tier 0 canonical documents are present.
2. One current RCC per canonical document is present, with RCC::Target-SHA-256 matching the canonical document's Integrity::SHA-256.
3. All five Tier 3 stewardship documents are present.
4. A LICENSE file is present at the bundle root containing the CC BY-ND 4.0 legal text.
5. Cross-document dependency references are consistent: where artifact A declares a dependency on artifact B at a specific issuance, the bundle contains artifact B at that issuance.

Any missing artifact or inconsistent cross-reference results in **release rejection**.

---

## Phase 9 — Cryptographic Signing

After the release manifest has been generated:

1. Validator SHALL confirm the release manifest contains a SHA-256 entry for every artifact in the bundle.
2. The release manifest SHALL be signed using a PGP key whose fingerprint is listed as ACTIVE in SC-AUTHORSHIP-000001 §5.
3. The signature SHALL be a detached ASCII-armored file distributed alongside the release manifest.
4. Validator SHALL confirm the detached signature is present and verifiable against the published fingerprint.

Absence of a valid signature results in **release rejection**.

---

# Release Approval Checklist

A release MAY be issued only if ALL of the following conditions are satisfied:

1. Every artifact with Release Status = ISSUED has passed Phases 1-7 (per-artifact validation).
2. The assembled bundle has passed Phase 8 (bundle completeness verification).
3. The release manifest has been generated and contains SHA-256 entries matching all artifacts.
4. The release manifest has been PGP-signed per Phase 9.
5. The export ZIP has been generated and its SHA-256 recorded in the ZIP manifest.
6. The ZIP manifest references the release manifest.

All six conditions are binary (pass/fail). Any failure halts release.
